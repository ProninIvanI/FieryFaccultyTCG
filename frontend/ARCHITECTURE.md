# Архитектура Frontend

Проект использует слоистую архитектуру (Layered Architecture) для обеспечения масштабируемости и поддерживаемости кода.

## Структура папок

```
src/
├── components/     # Переиспользуемые UI компоненты
├── pages/          # Страницы/экраны приложения
├── hooks/          # Кастомные React хуки
├── services/       # Сервисы и API слой
│   ├── api/        # API слой (axios instance, apiClient)
│   └── *.ts        # Бизнес-сервисы
├── utils/          # Вспомогательные утилиты
├── types/          # TypeScript типы и интерфейсы
├── constants/      # Константы приложения
├── styles/         # Глобальные стили
├── assets/         # Статические файлы (изображения, иконки)
├── App.tsx         # Главный компонент приложения
└── main.tsx        # Точка входа
```

## Описание слоев

### 1. **components/** - Компоненты
Переиспользуемые UI компоненты.

**Принципы:**
- Один компонент = одна папка
- Каждый компонент имеет свой модульный CSS файл
- Экспорт через `index.ts` для удобного импорта
- Компоненты должны быть переиспользуемыми и независимыми

**Структура компонента:**
```
ComponentName/
├── ComponentName.tsx
├── ComponentName.module.css
└── index.ts
```

**Примеры:**
- `Header` - заголовок страницы
- `Card` - карточка для контента
- `StatusBadge` - бейдж статуса

### 2. **pages/** - Страницы
Страницы/экраны приложения, которые состоят из компонентов.

**Принципы:**
- Одна страница = одна папка
- Страницы используют компоненты из `components/`
- Страницы могут использовать хуки из `hooks/`
- Экспорт через `index.ts`

**Структура страницы:**
```
PageName/
├── PageName.tsx
├── PageName.module.css
└── index.ts
```

**Примеры:**
- `HomePage` - главная страница

### 3. **hooks/** - Кастомные хуки
Переиспользуемая логика с состоянием.

**Принципы:**
- Инкапсулируют логику работы с данными
- Могут использовать сервисы из `services/`
- Возвращают состояние и функции для работы с ним
- Начинаются с префикса `use`

**Примеры:**
- `useHealthCheck` - проверка состояния backend

### 4. **services/** - Сервисы
Работа с внешними API и данными.

**Принципы:**
- Используют `services/api` (`apiClient`, `axiosInstance`) для HTTP запросов
- Возвращают типизированные данные
- Обрабатывают ошибки
- Не зависят от React
- Для auth-сценариев сервис остаётся единственной точкой входа для login/logout и работы с `fftcg_session`

**Примеры:**
- `healthService` - работа с health check API
- `authService` - login/logout, чтение и очистка клиентской сессии, отправка bearer token на backend
- `deckService` - загрузка и сохранение пользовательских колод через backend API
- `gameWsService` - типизированный WebSocket-клиент для live-PvP, который отправляет `join`, `roundDraft.replace`, `roundDraft.lock` и получает `state`, `roundDraft.snapshot`, `roundStatus`, `roundResolved`, `error`

### 5. **utils/** - Утилиты
Вспомогательные функции и утилиты.

**Принципы:**
- Чистые функции (pure functions)
- Без побочных эффектов
- Переиспользуемость
- Независимость от React

**Примеры:**
- `index.ts` - переэкспорт утилит

### 6. **types/** - Типы TypeScript
Общие типы и интерфейсы для всего приложения.

**Принципы:**
- Переиспользуемые типы
- Типы для API ответов
- Типы для компонентов (props, state)

**Примеры:**
- `ApiResponse<T>` - стандартный формат ответа API
- `HealthCheckResponse` - тип для health check

### 7. **constants/** - Константы
Константы приложения (URLs, маршруты, конфигурация).

**Принципы:**
- Все константы в одном месте
- Использование `as const` для типобезопасности
- Группировка по назначению

**Примеры:**
- `API_URL` - базовый URL API
- `ROUTES` - маршруты приложения

### 8. **styles/** - Глобальные стили
Глобальные CSS стили и CSS переменные.

**Принципы:**
- CSS переменные для темизации
- Глобальные стили для базовых элементов
- Модульные стили для компонентов в их папках

**Примеры:**
- `index.css` - глобальные стили и CSS переменные

### 9. **assets/** - Статические файлы
Изображения, иконки, шрифты и другие статические ресурсы.

**Принципы:**
- Организация по типам файлов
- Оптимизация для production

## Поток данных

```
User Interaction
    ↓
pages/           (страница обрабатывает событие)
    ↓
hooks/           (кастомный хук с логикой)
    ↓
services/        (бизнес-сервис, например healthService)
    ↓
services/api/apiClient    (типизированный API клиент)
    ↓
services/api/axiosInstance (axios с interceptors)
    ↓
Backend API
    ↓
Response
    ↓
services/api/axiosInstance (обработка ответа в interceptor)
    ↓
services/api/apiClient    (форматирование ответа)
    ↓
services/        (обработка данных в бизнес-сервисе)
    ↓
hooks/           (обновление состояния)
    ↓
pages/           (обновление UI)
    ↓
components/      (рендеринг компонентов)
```

### PvP-сценарий во frontend

- `PlayPvpPage` не собирает матч локально и не хранит собственный source of truth для game state.
- Экран получает список колод через `deckService`, выбирает `deckId` и передаёт его в `gameWsService.joinSession(...)`.
- После server `state` экран строит derived UI-представления поверх snapshot: match spotlight, локальный player snapshot, линии стола `соперник/ты`, event feed, основные размеры зон и руку локального игрока.
- `PlayPvpPage` держит только локальное UX-состояние поверх hidden-round draft:
  - выбор карты или закреплённого объекта;
  - удаление действия из боевой ленты;
  - выбор цели;
  - lock-in;
  - восстановление собственного draft после reconnect через `roundDraft.snapshot`.
- PvP presentation-layer оформлен как единая tabletop-сцена: player/deck widgets собраны в один общий вертикальный side-column в порядке `оппонент -> колода соперника -> твоя колода -> ты`, а центральный arena-frame и hand zone собираются только из derived UI-данных и не становятся source of truth.
- Линии существ на поле намеренно компактные и горизонтальные, а рука локального игрока занимает нижнюю центральную зону арены; эти visual affordance помогают читать ход и темп матча, но не добавляют новой игровой логики поверх server snapshot.
- Карты в руке и боевой ленте больше не ограничиваются именем и `instanceId`: они резолвят тип, школу, стоимость, эффект и summon-характеристики (`HP / ATK / SPD`) через shared catalog flow, чтобы PvP-визуал оставался согласованным с `DeckPage`.
- Имена карт для руки и существ на столе резолвятся через `definitionId` из локального каталога `cards.json`, но принадлежность сущностей зонам и доступность действий берётся только из server snapshot.
- `Raw state` убран в debug-блок, чтобы не быть основным UX-слоем матча и при этом оставаться доступным для отладки.
- Текущий round-flow уже выведен в интерфейс:
  - локальный игрок работает с единой боевой лентой вместо отдельной панели `черновика раунда`;
  - клик по карте из руки сразу добавляет `Summon`, target-спелл или `PlayCard` в ленту;
  - действия закреплённых объектов конфигурируются inline в карточке ленты;
  - UI показывает placement и preview слоя резолва, но не обещает FIFO-исполнение;
  - после `roundResolved` результат связывается с локальными intent по `intentId`;
  - для последнего резолва UI показывает `resolved/fizzled`, reason code, summary и reveal timeline в фактическом server-side порядке.
- Draft и post-round presentation больше не опираются на сырой `targetId`: PvP UI резолвит человекочитаемые target label’ы (`Твой маг`, `Маг user_2`, `Твое существо ...`) из актуального snapshot и использует их в очереди, snapshot restore и timeline.
- Правой contextual-панели в основном PvP UX больше нет: action/target indicators живут прямо в карточках боевой ленты, а смена цели происходит кликом по подсвеченным сущностям на поле.
- PvP UI всё ещё остаётся минимальным playable-срезом: bootstrap ручной, lobby/matchmaking и ручной smoke в двух реальных клиентах остаются следующими итерациями.

### Visual guardrails для PvP-экрана

- PvP-экран проектируется от viewport, а не от бесконечной страницы: базовое боевое состояние на desktop должно помещаться по высоте без общего vertical page-scroll.
- При нехватке места сначала уменьшается UI-плотность, а не растягивается сцена:
  - компактнее становятся боковые панели, HUD, отступы, размеры карт и вторичные подписи;
  - декоративные блоки и воздух ужимаются раньше, чем игрок теряет целостный обзор поля.
- Игрок должен видеть одновременно статус раунда, поле, свою руку и основные действия текущего решения; если один из этих слоёв уезжает за экран, layout считается неудачным.
- Горизонтальные scroller’ы допустимы только как локальный компромисс для насыщенных зон (`рука`, `боевая лента`, `линия существ`), но:
  - они не должны быть единственным способом читать базовое состояние матча;
  - первая и последняя карточки не должны резаться контейнером;
  - hover/selection состояния не должны ломать читаемость соседних карточек.
- Контраст — часть архитектуры интерфейса, а не косметика:
  - служебные подписи, статусы и empty-state не должны использовать слабочитаемые пары цветов;
  - визуальный акцент не должен спорить с основным текстом за внимание.
- Все layout-решения PvP должны проходить ручную smoke-проверку в живом браузере хотя бы на одном desktop viewport до того, как решение считается приемлемым.

### Текущая round-модель в PvP UI

- PvP UI уже работает вокруг скрытого раунда, а не вокруг одиночного немедленного action dispatch.
- Локальный игрок собирает единую боевую ленту:
  - элементы читаются слева направо;
  - боевой порядок берётся из core-derived placement;
  - лента может содержать закреплённые `BoardItem`, действия из руки и действия закреплённых объектов.
- До lock-in содержимое очереди соперника скрыто:
  - UI показывает только факт готовности соперника через `roundStatus`;
  - собственный draft восстанавливается персональным `roundDraft.snapshot`;
  - скрытая информация не раскрывается через derived UI, debug labels или состояние WebSocket-клиента.
- После подтверждения обоих игроков UI показывает результат раунда как уже рассчитанный server/game-core resolution, а не как локальную симуляцию.

### Как UI должен трактовать порядок карт

- Визуальный порядок элементов в боевой ленте не равен гарантированному FIFO-порядку применения.
- UI обязан объяснять игроку две сущности:
  - `боевая лента` — как `game-core` расположил объекты и действия в текущем derived view;
  - `слой резолва` — когда действие реально будет обработано движком.
- Для этого ribbon-entry должен уметь показывать:
  - тип действия;
  - целевой слой резолва;
  - цель действия;
  - статус `draft / locked / resolved / fizzled`, когда такая модель появится.
- Если две карты визуально стоят рядом, но относятся к разным слоям, frontend не должен обещать пользователю прямой FIFO-резолв.

### Граница ответственности frontend

- Frontend может:
  - дать игроку выбрать карту, действие и цель внутри боевой ленты;
  - подсвечивать возможные цели и стоимость;
  - визуально маркировать защитные, атакующие и summon-действия;
  - восстанавливать собственный draft после reconnect;
  - связывать результат `roundResolved` с локальными intent по `intentId`.
- Frontend не должен:
  - сам вычислять, какая карта сработает раньше в итоговом раунде;
  - сам решать спорные tie-break кейсы;
  - менять outcome на основе локального порядка, если `game-core` решил иначе.
- Если UI показывает прогноз порядка, этот прогноз должен быть derived из правил `game-core`, а не из ad-hoc frontend логики.

### Ближайшие UI-задачи

- расширить battle log / reveal timeline от текущего минимального слоя до более подробного post-round UX;
- пройти документацию и сопутствующие архитектурные заметки после завершения UX-перехода;
- пройти ручной smoke round-flow в двух реальных клиентах.

### Recent PvP UI Notes

- `PlayPvpPage` resolves side-column player portraits from the selected `characterId` in match/deck state instead of static placeholders.
- The opponent hand is rendered as a mirrored top fan of card backs, so the UI communicates hand size without exposing hidden card content.

### Shared Catalog Flow

- Нормализация карточного каталога вынесена в `game-core/src/cards/catalog.ts` и используется как единый источник данных для frontend и server.
- `PlayPvpPage`, `CardsPage` и `DeckPage` больше не парсят `cards.json` каждая по-своему: они получают либо `normalizeCatalog(...)`, либо уже готовые summary-builder’ы из `game-core`.
- Для каталога централизованы:
  - metadata-типизация карточек и персонажей;
  - нормализация `cards + characters`;
  - summary-builder’ы для UI-экранов;
  - label/helper’ы для школ и карточных типов.
- Такой поток уменьшает риск расхождения между PvP UI, каталогом карт, декбилдером и server contract.

### Зафиксированный техдолг

- Сценарий `PlayPvpPage` в тестах стабилизирован через управляемую загрузку колод, явные `act(...)`-границы и router `future` flags, поэтому `act(...)` и React Router warnings больше не считаются активным техдолгом.
- Следующий техдолг по PvP UI уже не про шум тестов, а про расширение action-flow, углубление reveal UX и ручной smoke-сценарий в двух реальных клиентах.

## Пример добавления новой функциональности

Допустим, нужно добавить страницу со списком пользователей:

1. **types/index.ts** - добавить интерфейс `User`
2. **services/api/** - API слой уже настроен, можно добавить специфичные interceptors если нужно
3. **services/userService.ts** - методы для работы с API пользователей (использует `apiClient` из `services/api`)
4. **hooks/useUsers.ts** - хук для получения и управления списком пользователей
5. **components/UserList/UserList.tsx** - компонент списка пользователей
6. **components/UserCard/UserCard.tsx** - компонент карточки пользователя
7. **pages/UsersPage/UsersPage.tsx** - страница со списком пользователей
8. **constants/index.ts** - добавить маршрут `/users`

## Преимущества такой архитектуры

✅ **Масштабируемость** - легко добавлять новые функции и компоненты  
✅ **Поддерживаемость** - четкое разделение ответственности  
✅ **Переиспользование** - компоненты и хуки можно использовать в разных местах  
✅ **Тестируемость** - каждый слой можно тестировать отдельно  
✅ **Читаемость** - понятная структура для новых разработчиков  
✅ **Типобезопасность** - TypeScript типы на всех уровнях

## Соглашения по именованию

- **Компоненты**: PascalCase (`UserCard.tsx`)
- **Хуки**: camelCase с префиксом `use` (`useHealthCheck.ts`)
- **Сервисы**: camelCase (`healthService.ts`)
- **Утилиты**: camelCase (`date.ts`, `format.ts`)
- **Типы**: PascalCase (`ApiResponse`)
- **Константы**: UPPER_SNAKE_CASE (`API_URL`)
- **Стили**: kebab-case для файлов (`header.module.css`)
