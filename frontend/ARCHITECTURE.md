# Архитектура Frontend

Проект использует слоистую архитектуру (Layered Architecture) для обеспечения масштабируемости и поддерживаемости кода.

## Структура папок

```
src/
├── components/     # Переиспользуемые UI компоненты
├── pages/          # Страницы/экраны приложения
├── hooks/          # Кастомные React хуки
├── services/       # Сервисы и API слой
│   ├── api/        # API слой (axios instance, apiClient)
│   └── *.ts        # Бизнес-сервисы
├── utils/          # Вспомогательные утилиты
├── types/          # TypeScript типы и интерфейсы
├── constants/      # Константы приложения
├── styles/         # Глобальные стили
├── assets/         # Статические файлы (изображения, иконки)
├── App.tsx         # Главный компонент приложения
└── main.tsx        # Точка входа
```

## Описание слоев

### 1. **components/** - Компоненты
Переиспользуемые UI компоненты.

**Принципы:**
- Один компонент = одна папка
- Каждый компонент имеет свой модульный CSS файл
- Экспорт через `index.ts` для удобного импорта
- Компоненты должны быть переиспользуемыми и независимыми

**Структура компонента:**
```
ComponentName/
├── ComponentName.tsx
├── ComponentName.module.css
└── index.ts
```

**Примеры:**
- `Header` - заголовок страницы
- `Card` - карточка для контента
- `StatusBadge` - бейдж статуса

### 2. **pages/** - Страницы
Страницы/экраны приложения, которые состоят из компонентов.

**Принципы:**
- Одна страница = одна папка
- Страницы используют компоненты из `components/`
- Страницы могут использовать хуки из `hooks/`
- Экспорт через `index.ts`

**Структура страницы:**
```
PageName/
├── PageName.tsx
├── PageName.module.css
└── index.ts
```

**Примеры:**
- `HomePage` - главная страница

### 3. **hooks/** - Кастомные хуки
Переиспользуемая логика с состоянием.

**Принципы:**
- Инкапсулируют логику работы с данными
- Могут использовать сервисы из `services/`
- Возвращают состояние и функции для работы с ним
- Начинаются с префикса `use`

**Примеры:**
- `useHealthCheck` - проверка состояния backend

### 4. **services/** - Сервисы
Работа с внешними API и данными.

**Принципы:**
- Используют `services/api` (`apiClient`, `axiosInstance`) для HTTP запросов
- Возвращают типизированные данные
- Обрабатывают ошибки
- Не зависят от React
- Для auth-сценариев сервис остаётся единственной точкой входа для login/logout и работы с `fftcg_session`

**Примеры:**
- `healthService` - работа с health check API
- `authService` - login/logout, чтение и очистка клиентской сессии, отправка bearer token на backend
- `deckService` - загрузка и сохранение пользовательских колод через backend API
- `gameWsService` - типизированный WebSocket-клиент для live-PvP, который отправляет `join/action` и получает server `state/error`

### 5. **utils/** - Утилиты
Вспомогательные функции и утилиты.

**Принципы:**
- Чистые функции (pure functions)
- Без побочных эффектов
- Переиспользуемость
- Независимость от React

**Примеры:**
- `index.ts` - переэкспорт утилит

### 6. **types/** - Типы TypeScript
Общие типы и интерфейсы для всего приложения.

**Принципы:**
- Переиспользуемые типы
- Типы для API ответов
- Типы для компонентов (props, state)

**Примеры:**
- `ApiResponse<T>` - стандартный формат ответа API
- `HealthCheckResponse` - тип для health check

### 7. **constants/** - Константы
Константы приложения (URLs, маршруты, конфигурация).

**Принципы:**
- Все константы в одном месте
- Использование `as const` для типобезопасности
- Группировка по назначению

**Примеры:**
- `API_URL` - базовый URL API
- `ROUTES` - маршруты приложения

### 8. **styles/** - Глобальные стили
Глобальные CSS стили и CSS переменные.

**Принципы:**
- CSS переменные для темизации
- Глобальные стили для базовых элементов
- Модульные стили для компонентов в их папках

**Примеры:**
- `index.css` - глобальные стили и CSS переменные

### 9. **assets/** - Статические файлы
Изображения, иконки, шрифты и другие статические ресурсы.

**Принципы:**
- Организация по типам файлов
- Оптимизация для production

## Поток данных

```
User Interaction
    ↓
pages/           (страница обрабатывает событие)
    ↓
hooks/           (кастомный хук с логикой)
    ↓
services/        (бизнес-сервис, например healthService)
    ↓
services/api/apiClient    (типизированный API клиент)
    ↓
services/api/axiosInstance (axios с interceptors)
    ↓
Backend API
    ↓
Response
    ↓
services/api/axiosInstance (обработка ответа в interceptor)
    ↓
services/api/apiClient    (форматирование ответа)
    ↓
services/        (обработка данных в бизнес-сервисе)
    ↓
hooks/           (обновление состояния)
    ↓
pages/           (обновление UI)
    ↓
components/      (рендеринг компонентов)
```

### PvP-сценарий во frontend

- `PlayPvpPage` не собирает матч локально и не хранит собственный source of truth для game state.
- Экран получает список колод через `deckService`, выбирает `deckId` и передаёт его в `gameWsService.joinSession(...)`.
- После server `state` экран строит только derived UI-представления: summary матча, локальный player snapshot, размеры зон, руку локального игрока.
- Имена карт для отображения руки сейчас резолвятся по локальному каталогу `cards.json`, но сама принадлежность карты руке и зонам берётся только из server snapshot.
- Это временный технический UI-слой для стабилизации PvP; полноценное игровое поле и action-panel остаются следующим фронтенд-этапом.

### Зафиксированный техдолг

- В тестах `PlayPvpPage` остаются `act(...)` warnings от асинхронных обновлений состояния вокруг загрузки колод и WS-событий.
- Это не блокирует текущую сборку, type-check, lint и прохождение тестов, поэтому техдолг считаем `important`, а не `critical`.
- Следующий проход по frontend должен либо стабилизировать тест через более явные `act/waitFor` границы, либо вынести часть асинхронной orchestration-логики страницы в более узкий hook/service.

## Пример добавления новой функциональности

Допустим, нужно добавить страницу со списком пользователей:

1. **types/index.ts** - добавить интерфейс `User`
2. **services/api/** - API слой уже настроен, можно добавить специфичные interceptors если нужно
3. **services/userService.ts** - методы для работы с API пользователей (использует `apiClient` из `services/api`)
4. **hooks/useUsers.ts** - хук для получения и управления списком пользователей
5. **components/UserList/UserList.tsx** - компонент списка пользователей
6. **components/UserCard/UserCard.tsx** - компонент карточки пользователя
7. **pages/UsersPage/UsersPage.tsx** - страница со списком пользователей
8. **constants/index.ts** - добавить маршрут `/users`

## Преимущества такой архитектуры

✅ **Масштабируемость** - легко добавлять новые функции и компоненты  
✅ **Поддерживаемость** - четкое разделение ответственности  
✅ **Переиспользование** - компоненты и хуки можно использовать в разных местах  
✅ **Тестируемость** - каждый слой можно тестировать отдельно  
✅ **Читаемость** - понятная структура для новых разработчиков  
✅ **Типобезопасность** - TypeScript типы на всех уровнях

## Соглашения по именованию

- **Компоненты**: PascalCase (`UserCard.tsx`)
- **Хуки**: camelCase с префиксом `use` (`useHealthCheck.ts`)
- **Сервисы**: camelCase (`healthService.ts`)
- **Утилиты**: camelCase (`date.ts`, `format.ts`)
- **Типы**: PascalCase (`ApiResponse`)
- **Константы**: UPPER_SNAKE_CASE (`API_URL`)
- **Стили**: kebab-case для файлов (`header.module.css`)
