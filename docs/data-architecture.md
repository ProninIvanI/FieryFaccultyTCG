# Архитектура Данных

Документ фиксирует, какие данные есть в проекте, где они должны храниться и какой слой считается источником истины.

## Цель

- убрать размазывание данных между `frontend`, `backend`, `server` и `game-core`;
- зафиксировать единый `source of truth` для каждого домена;
- подготовить основу для перехода от локального demo/mock-хранения к нормальному серверному хранению.

## Текущая ситуация

- Пользовательские данные уже начали переезжать в `backend + postgres`.
- `frontend` больше не хранит список пользователей как источник истины.
- В `localStorage` остаётся только текущая клиентская session info.
- `backend` уже начал быть источником истины для пользователей через `register/login/me`.
- `WS identity` уже начала опираться на backend-auth token, а не на доверенный клиентом `playerId`.
- `server` уже держит активные матчевые сессии в runtime, что нормально для live-PvP.

## Главные принципы

1. У каждого типа данных должен быть один владелец.
2. `frontend` не хранит критичные shared-данные как источник истины.
3. Всё, что должно жить между устройствами, браузерами и сессиями, хранится на сервере.
4. Всё, что влияет на игровой исход, остаётся в `game-core`.
5. `server` отвечает за live-runtime матчей, но не становится универсальной БД проекта.
6. Sensitive-данные не логируются и не хранятся в открытом виде.

## Домены Данных

### 1. Users / Auth

Что входит:
- пользователь;
- email;
- username;
- password hash;
- session / refresh token metadata;
- профиль и базовые настройки аккаунта.

Источник истины:
- `backend + postgres`

Почему:
- эти данные должны быть общими между браузерами и устройствами;
- они нужны для серверной идентичности и нормального login-flow;
- frontend-localStorage здесь годится только как временный demo/mock.

Frontend хранит:
- только текущую клиентскую сессию или access token;
- никаких списков пользователей.

Текущий статус:
- `backend + postgres` уже используются для `users/auth`;
- `server` уже связывает auth с WebSocket identity;
- следующий шаг в этом домене: добить logout/session invalidation и ручной staging-smoke.

### 2. Game Data / Справочники

Что входит:
- карты;
- персонажи;
- возможно правила, шаблоны эффектов и пресеты.

Источник истины:
- на текущем этапе `game-core` / data-driven файлы в репозитории

Почему:
- эти данные тесно связаны с игровой логикой;
- удобно версионировать их вместе с кодом;
- они должны быть согласованы с `game-core`.

Когда переносить в БД:
- только если появится реальная потребность в админском редактировании контента.

### 3. Live Match Runtime

Что входит:
- активная PvP-сессия;
- текущий сериализованный match state;
- привязка игроков к live-session;
- привязка socket -> user/player;
- seed и runtime-служебные метаданные матча.

Источник истины:
- `server`

Почему:
- это короткоживущие live-данные;
- они нужны для real-time оркестрации;
- их не обязан напрямую хранить frontend.

Важно:
- live-runtime не заменяет persistent-хранилище истории матчей.

### 4. Match History / Replay / Results

Что входит:
- завершённые матчи;
- replay;
- история команд;
- снимки state;
- результаты матчей;
- отладочная история по уже сыгранным матчам.

Источник истины:
- `backend + postgres`

Почему:
- это уже persistent-данные;
- они нужны для истории, аналитики, replay, дебага и админки.

### 5. User Decks

Что входит:
- сохранённые колоды;
- состав колоды;
- привязка колоды к пользователю;
- версия/статус колоды.

Источник истины:
- `backend + postgres`

Почему:
- колоды принадлежат пользователю;
- они должны быть доступны между устройствами и сессиями.

### 6. Content / Admin

Что входит:
- новости;
- debug/admin settings;
- тестовые серверные настройки;
- будущие административные сущности.

Источник истины:
- `backend + postgres`

Почему:
- это серверный управляемый контент;
- он не должен жить только во frontend.

### 7. UI State

Что входит:
- открытые/закрытые панели;
- локальные фильтры;
- технические client-side preferences;
- временное состояние формы.

Источник истины:
- `frontend`

Почему:
- это локальное состояние интерфейса;
- оно не должно утекать в backend, если не представляет пользовательскую сущность.

## Целевая Карта Хранения

- `frontend`
  - UI-state
  - временная клиентская session info / token

- `backend + postgres`
  - users
  - auth sessions / refresh tokens
  - decks
  - deck cards
  - news
  - matches
  - match players
  - match replays

- `server`
  - активные live-match sessions
  - runtime match state
  - socket bindings

- `game-core`
  - детерминированная логика
  - доменная модель матча
  - game-data справочники, пока они остаются файловыми

## Предлагаемая Минимальная Persistent-Схема

Первый realistic набор таблиц:
- `users`
- `auth_sessions` или `refresh_tokens`
- `decks`
- `deck_cards`
- `matches`
- `match_players`
- `match_replays`
- `news`

## Порядок Реализации

### Этап 1. Users / Auth

- backend-модель пользователя;
- хранение пользователей в `postgres`;
- `register/login/me`;
- убрать список пользователей из `frontend/localStorage`.

### Этап 2. Decks

- persistent-хранение колод;
- привязка колод к пользователю;
- загрузка колод через backend API.

### Этап 3. Auth -> WS Identity

- связать backend-auth с `playerId`;
- убрать доверие к произвольному `userId` из UI;
- привязать websocket к аутентифицированному пользователю.

### Этап 4. Match History / Replay

- сохранять матчи и replay;
- подготовить базу для анализа, истории и debug.

### Этап 5. Content / Admin

- новости;
- admin/debug settings;
- управляемые серверные сущности.

## Что Пока Не Делаем

- не тащим сразу сложную production-auth-систему;
- не переносим весь game-data контент в БД без реальной необходимости;
- не смешиваем live-runtime матчей и persistent-историю в одну сущность;
- не храним sensitive-данные во frontend как источник истины.

## Первое Практическое Решение

Первый рабочий вертикальный срез:
- `users` в `backend + postgres`;
- `register/login/me`;
- `frontend` перестаёт хранить список пользователей локально;
- текущая PvP-auth-связка готовится к переходу на server-backed identity.

Статус:
- реализовано частично и уже стало рабочей базой для следующего шага;
- текущий следующий фокус: `decks`, затем развитие persistent-пользовательских сущностей.
