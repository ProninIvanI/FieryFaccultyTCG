# PvP current fix plan

## Цель

Собрать актуальный список правок по PvP после smoke-проверки и вести его как рабочий документ: добавлять пункты, удалять неактуальные, отмечать выполненное и согласовывать порядок перед реализацией.

Правки делаем пакетами: сначала согласуем пункт, потом меняем код, затем выполняем кодовые проверки. Визуальный smoke после правок выполняется вручную пользователем.

## Текущий план

### 1. Вернуть нормальный card-shell для ленты

Статус: выполнено (2026-05-06).

Проблема:

- Карта в руке выглядит как нормальная единая карта.
- После выкладывания карты в ленту драфта элементы визуально распадаются: art, название, `SPD`, mana-флажок и кнопка удаления выглядят как отдельные части вне корпуса карты.
- Это регресс: раньше выложенная карта воспринималась как полноценная карта, а не как набор разрозненных элементов.

Что нужно сделать:

- Карта в ленте должна выглядеть как карта из руки: единый корпус, art, стоимость/скорость, название и управляющие элементы внутри рамки.
- Стабилизировать размеры карты, чтобы она не меняла высоту из-за действий или дополнительной информации.
- Кнопка удаления должна быть частью card-shell или компактным контролом, который не разрушает силуэт карты.

Исполнительский brief для AI-агента:

- Основная зона правок: `frontend/src/pages/PlayPvpPage/LocalBattleRibbon.tsx` и `frontend/src/pages/PlayPvpPage/PlayPvpPage.module.css`.
- Референс для внешнего вида: `LocalHandTray.tsx` + CSS-классы `handCard`, `handCardArtwork`, `handCardBody`, `handCardTitle`, `handManaGem`, `handStatPill`.
- Цель: action-card в ленте должен стать полноценной картой, а не набором отдельных блоков.
- Не менять игровую логику, `targetType`, резолв, порядок действий и отправку roundDraft.
- Не трогать card-shell существ и replay-карточки сверх минимально необходимого, если они не ломаются из-за общей CSS-структуры.

Ожидаемая структура action-card:

- Один внешний card-shell с фиксированной/стабильной высотой.
- Внутри card-shell:
  - art-зона;
  - mana badge/gem;
  - название карты;
  - `SPD`/скорость;
  - компактная кнопка удаления или remove-контрол на рамке карты;
  - target tabs/флажки цели как overlay, но в пределах card-shell.
- Никакие обязательные элементы не должны находиться визуально ниже карты или отдельно от неё.

Подход к реализации:

1. Сравнить текущую разметку action-card в `LocalBattleRibbon.tsx` с разметкой карты в руке.
2. Либо вынести небольшой общий presentational card-shell, либо точечно привести `ribbonCardAction` к структуре `handCard`.
3. Перенести title, `SPD`, mana и remove-control внутрь card-shell.
4. Для target tabs оставить компактный overlay/флажок внутри карты, не расширяющий высоту.
5. Убрать визуальную зависимость от внешней кнопки “Убрать из ленты” под картой.

Критерии готовности:

- На smoke-скрине выложенная карта выглядит как одна цельная карта.
- Название, `SPD`, mana и remove-control находятся внутри единой рамки.
- Карта в ленте и карта в руке выглядят как родственные варианты одного UI.
- При 1, 2 и 3+ картах в ленте размер card-shell остаётся стабильным.
- Target tabs не ломают силуэт карты и не выталкивают содержимое наружу.

Проверки для этого пакета:

- `npm.cmd --prefix frontend run type-check`
- `npm.cmd --prefix frontend run lint`
- `npm.cmd --prefix frontend run build`
- Желательно: `npm.cmd --prefix frontend run test -- PlayPvpPage.test.tsx`

### 2. Развести карту и служебные действия

Статус: выполнено (2026-05-06).

Проблема:

- Карта начинает выглядеть как панель управления, а не как игровая карта.

Что нужно сделать:

- Действия не должны менять высоту карты и не должны растягивать layout.

### 4. Вернуть понятные target-флажки на поле

Статус: выполнено (2026-05-06).

Проблема:

- При выборе карты/действия не все допустимые цели выглядят доступными.
- Флажок цели должен показывать сущности, которые можно атаковать или выбрать целью, но сейчас это поведение неочевидно.

Что нужно сделать:

- Если выбрана карта или действие, все допустимые цели должны явно подсвечиваться.
- На вражеском маге и на призванных существах должны появляться понятные маркеры “можно выбрать целью”.
- Клик по NPC должен менять цель выбранной карты/действия так же, как клик по магу.
- Активная выбранная цель должна визуально отличаться от просто доступной цели.

### 5. Разобраться с `targetType` боевых карт

Статус: выполнено для direct single-target damage spells (2026-05-06).

Проблема:

- У части прямых damage-заклинаний в данных стоит `targetType: "enemyCharacter"`, поэтому они могут выбирать только мага.
- Игрок ожидает, что одиночное атакующее заклинание сможет выбрать либо мага, либо призванное существо, если существо стоит на поле.
- Во фронте уже есть поддержка `enemyAny`, где кандидатами становятся и маг, и вражеские существа.

Что нужно решить:

- Либо все одиночные прямые damage-заклинания должны получить `targetType: "enemyAny"`.
- Либо часть карт сознательно бьёт только мага, но тогда это должно быть явно отражено в описании и визуале.

Предварительное предложение:

- Прямые single-target damage spells перевести на `enemyAny`.
- Массовые эффекты, ловушки, глобальные debuff/control и специальные карты разбирать отдельно по смыслу эффекта.

Результат:

- Direct single-target `spellKind: "damage"` карты переведены на `targetType: "enemyAny"`.
- Массовые эффекты, ловушки, DoT/displacement и специальные карты оставлены без изменения для отдельного разбора.
- `game-core` валидация реальной damage-карты против вражеского существа покрыта тестом.

## Smoke update: card-shell still not fixed

Дата: 2026-05-06.

Статус: исправлено в текущем пакете правок (2026-05-06), ждёт ручного visual smoke.

Наблюдение:

- Выложенная карта всё ещё не выглядит как полноценная карта.
- Визуально изменился только верхний блок/art и внешняя рамка.
- Название карты, `SPD` и кнопка “Убрать из ленты” остаются вне единого корпуса карты.
- Mana-флажок сбоку выглядит отдельным элементом, а не частью цельной карты.
- На примере “Лавовый поток” видно, что card-shell не стал единым: карта сверху, а вся важная информация живёт ниже отдельными строками.

Что должно измениться:

- В ленте должна рендериться полноценная карта, а не “art-блок + подписи снизу”.
- Название, скорость, стоимость/флажок и кнопка удаления должны быть внутри одного card-shell.
- Выложенная карта должна быть визуально близка к карте в руке по структуре: корпус, art-зона, нижняя информационная зона, служебные бейджи внутри рамки.
- Высота карты должна быть стабильной и не зависеть от дополнительных действий.
- Кнопка удаления не должна висеть отдельной кнопкой под картой; её нужно сделать компактным контролом внутри карты или на её рамке.

Критерии готовности:

- По скрину выложенной карты сразу видно один цельный прямоугольный card-shell.
- Ни один обязательный элемент карты не находится за пределами рамки.
- При одной карте и при нескольких картах в ленте силуэт карты остаётся одинаковым.
- Карта в ленте и карта в руке выглядят как варианты одного UI-компонента, а не как две разные сущности.

Ближайший пакет реализации:

1. Найти текущий рендер action-card в `LocalBattleRibbon`.
2. Сравнить DOM/CSS с `LocalHandTray`.
3. Либо переиспользовать общий card-shell, либо привести `ribbonCardAction` к той же внутренней структуре.
4. Перенести title/SPD/remove/mana внутрь card-shell.
5. Проверить, что изменения не ломают карточки существ и replay-карточки.

### R1. Attack target must be validated as enemy in `game-core`

Приоритет: P1.

Файл:

- `game-core/src/rounds/validateRoundDraft.ts`

Проблема:

- Фронт теперь может отправлять атаку существа по NPC как `targetType: "creature"`.
- В `validateRoundDraft` для `Attack` сейчас проверяется только существование цели через `validateTarget`.
- Такой контракт позволяет вручную отправить payload атаки по своему существу, если обойти UI.
- Правило “атаковать можно только вражескую сущность” должно проверяться в `game-core`.

Что нужно сделать:

- В ветке `intent.kind === "Attack"` после проверки source creature добавить core-валидацию цели.
- Цель атаки должна быть:
  - вражеским персонажем; или
  - вражеским существом.
- Атака по своему персонажу, своему существу, отсутствующей цели и неизвестному типу должна возвращать validation error.
- Не полагаться на фронтовый `isSelectableTarget`.

Критерии готовности:

- `validateRoundDraft` отклоняет Attack по своему существу.
- `validateRoundDraft` отклоняет Attack по своему магу.
- `validateRoundDraft` принимает Attack по вражескому существу.
- `validateRoundDraft` принимает Attack по вражескому магу.

Тесты:

- Добавить/обновить `game-core/tests/validate-round-draft.test.ts`.
- Минимально покрыть:
  - Attack по enemy creature проходит;
  - Attack по ally creature отклоняется.

Проверки:

- `npm.cmd --prefix game-core run test -- validate-round-draft.test.ts create-initial-round-intent.test.ts`

Результат:

- Исправлено.
- `npm.cmd --prefix game-core run type-check` — пройдено.
- `npm.cmd --prefix game-core run test -- validate-round-draft.test.ts create-initial-round-intent.test.ts` — пройдено.

### R3. Hover popover must not disappear between flag and panel

Приоритет: P2.

Файл:

- `frontend/src/pages/PlayPvpPage/PlayPvpPage.module.css`

Проблема:

- `.ribbonActionPopover` открывается через hover/focus на `.ribbonActionStack`.
- Сейчас popover начинается на `top: calc(100% + 6px)`.

- Убрать hover gap одним из способов:

- Не увеличивать высоту карты и не сдвигать соседние карты.

Критерии готовности:

- Можно перевести курсор с флажка на popover без схлопывания.
- Popover всё ещё не перехватывает клики, пока закрыт.
- Layout карты остаётся стабильным.

Проверки:

- `npm.cmd --prefix frontend run test -- PlayPvpPage.test.tsx`
- Ручной visual smoke пользователем: hover по флажку действий существа.

Результат:
