# План развития `Ленты матча`

Дата: 2026-04-09
Статус: draft for implementation
Назначение: зафиксировать целевую модель `Ленты матча`, чтобы развивать её поэтапно без смешения с replay-полосой и live-draft UI.

## 1. Цель

Превратить текущую `Ленту матча` из плоского списка последних технических событий в читаемую историю матча:

- с группировкой по раундам;
- с возможностью раскрывать каждый раунд отдельно;
- с человекочитаемыми описаниями действий;
- с явным указанием, кто именно выполнил действие;
- с более подробным описанием результата действия и связанных эффектов.

Главный UX-принцип:

- `Лента матча` не заменяет replay-strip;
- `Лента матча` даёт текстовую историю матча;
- replay-strip остаётся визуальной сценой последнего резолва;
- live-draft остаётся интерфейсом текущего принятия решений.

## 2. Что есть сейчас

Текущее состояние во фронтенде:

- `Лента матча` строится из `state.log`;
- берутся только последние записи;
- они показываются плоским списком;
- текст событий слишком общий: `Action`, `Урон`, `Эффект`, `Призыв`;
- нет группировки по раундам;
- нет раскрытия/сворачивания;
- нет богатого описания `кто -> что -> в кого -> чем закончилось`.

Техническое ограничение текущих данных:

- `GameLogEntry` не содержит явного `roundNumber`;
- зато `roundResolved` уже содержит:
  - `roundNumber`;
  - фактический порядок `orderedActions`;
  - `playerId`;
  - `kind`;
  - `source`;
  - `target`;
  - `status`;
  - `reasonCode`;
  - `summary`;
  - `definitionId`.

Вывод:

- первая сильная версия `Ленты матча` должна строиться прежде всего от истории `roundResolved`;
- `state.log` можно подключать позже как дополнительный слой детализации.

## 3. Целевая модель

Нужен новый клиентский формат:

```ts
interface MatchFeedRoundSummary {
  roundNumber: number;
  title: string;
  subtitle: string;
  entries: MatchFeedEntrySummary[];
}

interface MatchFeedEntrySummary {
  id: string;
  actorLabel: string;
  actionLabel: string;
  targetLabel?: string;
  outcomeLabel: string;
  detailText?: string;
  tone: 'neutral' | 'success' | 'warning' | 'danger';
}
```

Смысл:

- один блок = один завершённый раунд;
- внутри блока лежат все шаги фактического резолва этого раунда;
- каждая запись описывает действие понятным текстом, а не техническим названием event type.

## 4. Источник данных

### 4.1. Первый этап

Источником правды делаем историю `roundResolved`, накопленную на фронтенде.

Нужно добавить на странице PvP:

- `resolvedRoundHistory: RoundResolutionResult[]`;
- пополнение истории на каждый новый `roundResolved`;
- защиту от дублей по `roundNumber`.

Это даст:

- точную группировку по раундам;
- правильный порядок действий внутри раунда;
- стабильный источник для текстовой истории.

### 4.2. Второй этап

После базовой версии можно расширять каждый round block дополнительными событиями из `state.log`:

- `damage`;
- `effect`;
- `summon`;
- другие полезные технические события.

Но только после ответа на вопрос:

- как надёжно соотносить эти события с конкретным раундом, если у `GameLogEntry` нет `roundNumber`.

Если без костылей это не делается, лучше:

- либо не использовать `state.log` для round grouping;
- либо позже аккуратно расширить модель в `game-core`.

## 5. Целевой UI

### 5.1. Общая структура

`Лента матча` должна стать списком round sections:

- `Раунд 1`
- `Раунд 2`
- `Раунд 3`

Порядок:

- сверху самый свежий завершённый раунд;
- старые раунды ниже.

### 5.2. Заголовок раунда

В заголовке раунда показывать:

- номер раунда;
- общее число шагов;
- краткий summary по участникам, например: `3 твоих, 2 соперника`;
- индикатор раскрыт/свернут.

### 5.3. Содержимое раскрытого раунда

Внутри раунда каждая запись должна отвечать на вопросы:

- кто сделал действие;
- какое именно действие;
- по какой цели;
- чем действие закончилось;
- что важно знать о результате.

Примеры целевых текстов:

- `Ты разыграл заклинание "Огненный шар"`
- `Соперник разыграл "Сферу воды"`
- `Ты выполнил атаку существом ally_creature_1`
- `Цель: Маг kaktak`
- `Итог: сработало`
- `Итог: сорвалось`
- `Причина: цель исчезла до резолва`

### 5.4. Уровни подробности

У каждой записи желательно иметь:

- короткий текст для быстрого чтения;
- дополнительную строку с пояснением результата;
- при необходимости badge со слоем резолва;
- diagnostics-only детали отдельно, не в основном UX.

## 6. Формирование человекочитаемых записей

Для каждого `ResolvedRoundAction` нужен formatter.

### 6.1. Что должно попадать в formatter

- имя игрока;
- тип действия;
- название карты, если есть `definitionId`;
- источник действия;
- цель действия;
- `status`;
- `reasonCode`;
- `summary`.

### 6.2. Какой текст нужен

На первом этапе:

- не технический `CastSpell`;
- а пользовательский текст вида `разыграл заклинание "Огненный шар"`.

На втором этапе:

- можно сделать более богатые формулировки под конкретные action kinds.

## 7. Этапы реализации

### Этап 1. Базовая round feed версия

Сделать только на фронтенде:

- накопление `resolvedRoundHistory`;
- новый builder `MatchFeedRoundSummary[]`;
- группировку по раундам;
- accordion UI;
- подробные текстовые записи на базе `orderedActions`.

Это уже даст большой UX-выигрыш без изменений `game-core`.

### Этап 2. Полировка текста

Улучшить formatter:

- красивее называть действия;
- лучше описывать цели;
- лучше переводить `status` и `reasonCode`;
- добавить round summary в заголовок.

### Этап 3. Дополнительные события внутри раунда

Опционально:

- пробовать обогатить round block событиями из `state.log`;
- подключать только те события, которые можно надёжно привязать к конкретному раунду.

## 8. Что пока не делать

Чтобы не расползти задачу:

- не переносить replay-strip в `Ленту матча`;
- не смешивать `Ленту матча` с live-draft;
- не перестраивать `game-core`, пока не упрёмся в реальные ограничения данных;
- не пытаться сразу построить идеальный combat log для каждого микрособытия.

## 9. Риски и решения

### Риск 1. Нет `roundNumber` у `state.log`

Решение:

- на первом этапе не делать `state.log` источником round grouping;
- использовать историю `roundResolved`.

### Риск 2. Слишком сухой текст

Решение:

- завести явные formatter helpers для `ResolvedRoundAction`;
- не выводить сырые названия action types как финальный UX.

### Риск 3. Лента станет слишком длинной

Решение:

- round sections по умолчанию могут быть свернутыми;
- раскрывать последний завершённый раунд автоматически;
- старые раунды оставлять компактными.

## 10. Тестовый план

Нужно покрыть:

- накопление истории нескольких `roundResolved`;
- корректную группировку по `roundNumber`;
- правильный порядок шагов внутри раунда;
- человекочитаемые подписи для spell/summon/attack/play card;
- разворачивание и сворачивание round sections;
- отсутствие дублирования одного и того же round result.

2. заменить текущий `matchEvents` на round-based feed;

4. завести formatter для текстов действий;
5. покрыть это UI-тестами.
