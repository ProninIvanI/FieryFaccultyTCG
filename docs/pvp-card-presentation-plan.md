# План: PvP card presentation

Дата: 2026-04-14  
Статус: active working draft

## 1. Назначение

Зафиксировать поэтапный план переработки визуального представления PvP-сцены:

- стола как главной игровой сцены;
- верхней и нижней руки;
- скрытой активности соперника;
- карточного представления `compact -> inspect`.

План нужен как опорный документ перед дальнейшими правками layout и карточного UX.

## 2. Контекст

PvP-экран уже прошёл несколько итераций по теме, общей геометрии и распределению зон, но остаются проблемы именно в карточном представлении и ощущении стола как центральной сцены матча.

Текущие симптомы:

- верхняя и нижняя руки ещё не до конца ощущаются частью стола;
- на больших desktop-экранах местами появляется лишний воздух, особенно внизу;
- часть служебных подписей и HUD-подсказок перегружает сцену;
- `compact-card` и `detailed-card` пока смешаны в одном представлении;
- скрытая активность соперника должна опираться только на безопасные публичные данные;
- inspect / hover-preview паттерн ещё не оформлен как отдельный UX-слой.

Главный принцип:

- активный матч это first-class игровая сцена;
- базовый desktop-view не должен превращаться в длинную страницу;
- стол должен быть главным;
- подробная информация о карте должна появляться по запросу, а не постоянно шуметь внутри каждой маленькой карточки;
- приватная информация соперника не должна утекать во frontend.

## 3. Что уже зафиксировано

### 3.1. Layout-сцена

- верхняя рука соперника живёт в верхней части поля;
- нижняя рука игрока живёт у нижней границы;
- слева идёт вертикальная ось:
  - аватар соперника;
  - блок завершения хода;
  - аватар игрока;
- правая колонка остаётся сервисной и вторичной;
- скрытая зона соперника уже добавлена в сцену;
- кнопка просмотра диагностики больше не висит отдельным overlay над полем и встроена в блок управления ходом;
- верхняя и нижняя руки уже перераспределены ближе к границам арены относительно ранних версий экрана.

### 3.2. Публичный сигнал по скрытой активности

Для скрытой активности соперника используем только публичное состояние раунда:

- `locked`
- `draftCount`

Важно:

- `handSize` не подходит как источник для скрытой активности во время draft;
- в текущей модели карта не уходит из руки в момент составления хода;
- значит безопасный и корректный сигнал это именно `draftCount`, а не изменение размера руки.

Текущее состояние:

- базовая скрытая зона соперника уже подключена к публичному `draftCount`;
- фиктивные скрытые карты по умолчанию убраны;
- дальше нужно улучшать уже не источник данных, а сам визуальный язык этой зоны.

## 4. Главные проблемы

### 4.1. Сцена

- верхние зоны уже стали чище, но ещё недостаточно втянуты в стол;
- на больших экранах нижняя часть всё ещё иногда даёт слишком много пустого воздуха;
- отдельные UI-блоки читаются как самостоятельные панели, а не как части общей сцены.

### 4.2. Карты

- компактные карточки перегружены постоянной информацией;
- маленькая карточка пытается одновременно быть:
  - объектом на столе;
  - полноразмерной карточкой;
  - панелью подробностей;
- из-за этого страдает и чистота сцены, и читаемость.

### 4.3. Скрытая активность соперника

- нельзя показывать реальное содержимое чужого draft;
- нельзя фейковать количество карт;
- индикация должна строиться на реальном `draftCount`;
- анимация должна быть сдержанной и функциональной, а не декоративной;
- текущую базовую реализацию нужно упростить визуально и сильнее встроить в стол.

### 4.4. Inspect-pattern

- сейчас подробности карты частично вшиваются в компактную карточку;
- в перспективе нужно перейти к двухрежимной модели:
  - `compact`
  - `inspect`

## 5. Целевая UX-модель

### 5.1. Compact card

Это режим постоянного отображения на столе.

Compact-card должна показывать только главное.

Для существа:

- имя;
- 1 маркер типа или роли;
- `ATK / HP / SPD`

Для заклинания / эффекта:

- имя;
- мана;
- speed или короткий класс действия

Чего не должно быть в compact-card постоянно:

- длинного текста эффекта;
- подробных вторичных подписей;
- дублирующих бейджей;
- расширенных расшифровок.

### 5.2. Inspect card

Это режим просмотра по hover / focus / selection.

Inspect-card должна показывать:

- полное имя;
- тип;
- школу;
- ману;
- боевые значения;
- полный эффект;
- при необходимости цель, модификаторы и дополнительные свойства.

Принцип:

- сцена остаётся чистой;
- подробности появляются только по намерению игрока.

### 5.3. Hidden opponent prep

Скрытая активность соперника должна показывать:

- количество подготовленных действий;
- факт фиксации хода;
- изменение состояния без раскрытия содержимого.

Не должна показывать:

- названия карт;
- типы карт;
- цели;
- порядок конкретных действий соперника.

## 6. Итерационный план

### Итерация 4. Очистка верхней части сцены

Цель:

- убрать визуальный шум;
- упростить технические подписи;
- подтянуть верхнюю руку и скрытую активность к столу.

Шаги:

- убрать декоративные анимации, которые выглядят как баг;
- сократить заголовки верхних зон до минимума;
- убрать лишние поясняющие тексты;
- сделать верхние зоны ниже и ближе к игровому полю.

Статус:

- реализовано локально;
- убрана шумная декоративная анимация;
- сокращены лишние подписи в верхних зонах;
- верхняя рука и скрытая зона подтянуты ближе к столу;
- требует живой визуальной оценки перед финальной фиксацией.

### Итерация 5. Нижняя часть сцены и воздух

Цель:

- уменьшить мёртвый воздух внизу;
- собрать `поле -> боевая лента -> рука` в одну нижнюю сцену;
- убрать конфликт карточек с нижней границей.

Шаги:

- перераспределить нижний row-budget арены;
- сделать нижнюю боевую зону ближе к руке;
- уменьшить конфликт руки с нижней кромкой;
- усилить ощущение, что стол занимает центр.

Статус:

- реализовано локально;
- перераспределён нижний бюджет сцены;
- нижняя боевая зона собрана ближе к руке;
- ослаблен конфликт карточек с нижней границей;
- требует живой визуальной оценки перед финальной фиксацией.

### Итерация 6. Compact-card model

Цель:

- перестать пытаться уместить в маленькую карту всю информацию сразу;
- сделать карточку на столе компактной и стабильной.

Шаги:

- упростить локальные карты на столе;
- сократить постоянный текст эффекта;
- оставить только ключевые числовые показатели и имя;
- привести разные типы карточек к единому принципу плотности.

Ожидаемый результат:

- стол становится чище;
- карточки читаются быстрее;
- появляется база под отдельный inspect-слой.

### Итерация 7. Inspect / hover-preview

Цель:

- перенести подробности карты в отдельный режим просмотра.

Шаги:

- сделать отдельный preview-слой;
- выбрать базовый триггер:
  - hover;
  - focus;
  - selection;
- вынести туда подробные значения и полный текст эффекта;
- сохранить стол чистым в базовом состоянии.

Ожидаемый результат:

- компактная сцена;
- подробная информация доступна без перегруза поля.

### Итерация 8. Живая скрытая активность соперника

Цель:

- сделать верхнюю скрытую активность соперника живой, но честной и безопасной.

Шаги:

- привязать количество слотов к `draftCount`;
- различать `drafting` и `locked`;
- добавить мягкие переходы при изменении количества действий;
- не раскрывать содержимое.

Ожидаемый результат:

- видно, что соперник действительно действует;
- нет утечки приватной информации;
- верх сцены ощущается живым.

Текущий статус:

- базовый технический фундамент уже есть в ветке;
- скрытая зона уже строится от `draftCount`, а не от фиктивных данных;
- сама визуальная модель ещё не финальная и нуждается в дальнейшей полировке после `Итерации 6` и `Итерации 7`.

## 7. Практический приоритет

Рекомендуемый порядок:

1. Визуально подтвердить `Итерацию 4`
2. Визуально подтвердить `Итерацию 5`
3. Перейти в `Итерацию 6`
4. Затем строить `Итерацию 7`
5. И только после этого полировать `Итерацию 8`

Причина:

- без чистой сцены хороший inspect не спасёт;
- без compact-card inspect будет просто заплаткой на перегруженную карточку;
- без публичного сигнала по `draftCount` скрытая активность соперника начинает врать.

## 8. Ограничения

### 8.1. Безопасность и приватность

- соперник не должен получать содержимое чужой руки;
- соперник не должен получать содержимое чужого draft;
- публично допустимы только безопасные derived-сигналы:
  - `locked`
  - `draftCount`
  - публичное состояние стола после резолва

### 8.2. Layout

- normal desktop match state без вертикального page-scroll;
- ключевые элементы матча должны читаться одновременно:
  - стол;
  - рука;
  - зона текущего решения;
  - статус раунда

### 8.3. Card UX

- compact-card не должна быть мини-версией полной карточки один в один;
- inspect не должен ломать layout стола;
- hover-preview не должен перекрывать критичные игровые controls.

## 9. Следующий практический шаг

Следующим кодовым этапом логично считать начало `Итерации 6`, потому что:

- `Итерация 4` уже локально очищает верх сцены;
- `Итерация 5` уже локально собирает нижнюю часть сцены;
- базовый фундамент скрытой активности соперника уже заведён на реальный `draftCount`.

Дальше нужно:

- определить финальный набор данных для compact-card;
- отдельно определить набор данных для inspect-card;
- зафиксировать, какая информация должна жить:
  - в руке;
  - в локальной боевой ленте;
  - в скрытых картах соперника.
