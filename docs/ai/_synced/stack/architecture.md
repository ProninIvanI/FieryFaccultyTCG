# Архитектура стека web-ccg

Версия: 1.0  
Дата: 2026-03-15

## 1. Общее описание

- Проект «Академия Ремесла» — веб‑прототип коллекционной карточной игры.
- Ядро игры вынесено в детерминированный `game-core` (TypeScript):
  - детерминированный RNG (seedable);
  - Event System, Command Pattern, State Machine;
  - полностью сериализуемое состояние матча.
- Сервер (Node.js + TypeScript + WebSocket) управляет:
  - сессиями матчей;
  - подключениями игроков;
  - маршрутизацией команд от клиентов в `game-core`;
  - логированием, реплеями и симуляциями.
- Фронтенд (React 18 + TypeScript + Vite):
  - тонкий UI‑слой над `game-core` и сервером;
  - предоставляет интерфейсы для:
    - запуска матчей;
    - настройки колод;
    - просмотра карт и новостей;
    - дебага и анализа матчей.

---

## 2. Слои и модули бэкенда

- `transport/ws/`:
  - инициализация WebSocket‑сервера;
  - управление подключениями, жизненным циклом соединений;
  - преобразование сырого трафика в команды/DTO.
- `application/`:
  - обработчики команд (application services);
  - оркестрация сессий и матчей;
  - вызовы `game-core` и сохранение результатов.
- `domain/game/`:
  - адаптеры, связывающие чистый `game-core` с инфраструктурой (логирование, персистенция);
  - правила хранения текущего состояния матча, seed и истории команд.
- `infrastructure/`:
  - хранение (БД/файлы для логов/реплеев, если используются);
  - конфигурация, logger, metrics.

---

## 3. Слои и модули фронтенда

- `pages/`:
  - маршруты и сборка экранов из контейнеров/виджетов;
  - соответствуют основным сценариям (матч, декбилдер, просмотр карт, новости, дебаг).
- `widgets/` или `features/`:
  - крупные части интерфейса: декбилдер, панель матча, список карт и т.п.
- `entities/`:
  - базовые UI‑представления доменных сущностей (карта, колода, матч, игрок).
- `shared/`:
  - переиспользуемые компоненты (кнопки, layout, модалки) и утилиты.

Фронтенд:
- не содержит игровой логики;
- работает с состоянием через:
  - ответы сервера по WebSocket;
  - публичный API `game-core` (команды, события, сериализованный state).

---

## 4. Принципы детерминизма

- Итог матча определяется только:
  - начальными данными (колоды, параметры матча);
  - начальным seed;
  - последовательностью входящих команд;
  - версией `game-core`.
- Сервер и клиент не добавляют собственный рандом, влияющий на результат:
  - рандом для визуальных эффектов разрешён только во фронтенде и не влияет на state;
  - все игровые случайные решения происходят внутри `game-core`.

---

## 5. Реплеи, симуляции и логирование

- Для каждого матча хранится:
  - seed/seed state;
  - текущий сериализованный state из `game-core`;
  - история команд.
- Реплей:
  - либо пересчитывает матч с начального состояния и команд;
  - либо восстанавливает сохранённый state + seed.
- Логи:
  - фиксируют ключевые события (создание/завершение матча, ошибки);
  - не содержат секретов и избыточных персональных данных.

---

## 6. Текущая модель PvP-раунда

- PvP уже работает в модели `simultaneous resolution combat`, а не в публичной turn-based схеме с мгновенным `action`.
- Раунд собирается из двух независимых player intent’ов:
  - каждый игрок скрыто собирает свою очередь действий;
  - до фиксации раунда игроки не видят содержимое очереди соперника;
  - после lock-in обе очереди передаются в `game-core` на детерминированный резолв.
- UI-порядок и порядок резолва остаются разными сущностями:
  - игрок собирает свою линию действий слева направо как намерение;
  - итоговый порядок применения эффектов рассчитывается только внутри `game-core` по слоям резолва и tie-break правилам.

### 6.1. Что считается source of truth

- `frontend` отвечает только за ввод намерения игрока:
  - выбор карт;
  - визуальный порядок действий в очереди;
  - reorder/remove;
  - подтверждение готовности к резолву;
  - показ результата после server/core resolution.
- `server` отвечает за:
  - приём и валидацию DTO;
  - хранение скрытых draft-очередей по игрокам;
  - синхронизацию lock-in обоих игроков;
  - передачу собранного раунда в `game-core`;
  - персональные snapshot-сообщения для восстановления собственного draft;
  - рассылку результата раунда клиентам.
- `game-core` остаётся единственным местом, где определяется:
  - допустимость действий;
  - принадлежность действия к слою резолва;
  - tie-breaker внутри слоя;
  - итоговый порядок применения эффектов и обновление game state.

### 6.2. Текущий lifecycle раунда

- Раунд проходит через состояния:
  1. `draft`
  2. `locked_waiting`
  3. `resolving`
  4. `resolved`
- До `lock-in` игрок может свободно менять свой draft.
- Когда locked только один игрок, матч остаётся в `locked_waiting`.
- Резолв стартует только после lock обоих игроков.
- После резолва `game-core` сохраняет `lastResolution`, переключает инициативу и открывает следующий раунд.

### 6.3. Порядок слоёв резолва

- Слои задаются явно и обрабатываются детерминированно:
  1. призыв существ;
  2. защитные модификаторы;
  3. защитные заклинания;
  4. прочие модификаторы;
  5. атакующие и контроль-заклинания;
  6. атаки существ и персонажей;
  7. конец раунда: cleanup, уменьшение длительностей, удаление погибших сущностей, пост-эффекты.
- Внутри одного слоя должен сохраняться предсказуемый порядок:
  - сначала по явному правилу priority/speed, если оно есть у действия;
  - затем по порядку действия в очереди игрока;
  - затем по инициативе раунда;
  - затем по детерминированному tie-breaker, не зависящему от UI.

### 6.3.1. Текущий source of truth для карточной механики

- Боевые свойства карты описываются в общем каталоге `game-core/data/cards.json`, а не выводятся из UI-текста или локальных frontend-словарей.
- Для боевой логики карта должна иметь явные поля:
  - `targetType`;
  - `resolutionRole`;
  - structured `effects[]`;
  - для призывов — реальные `hp` / `attack`.
- `game-core` использует эти поля как основной источник истины для:
  - валидации цели;
  - вычисления слоя резолва;
  - исполнения эффекта;
  - построения round-order внутри draft/resolve pipeline.
- Эвристики по типу “угадать роль карты по тексту эффекта” или “вывести статы существа из mana cost” считаются legacy fallback и не должны развиваться дальше.

### 6.4. Правила текущей реализации

- Скрытый выбор действий обязателен: соперник не должен видеть первую волну карт до начала резолва.
- Пока персонажи в runtime-стейте индексируются через выбранный `characterId`, матч не допускает двух игроков с одним и тем же магом: сервер обязан отклонять второй `join` заранее с явной причиной, а не доводить матч до `roundDraft`-ошибок вида `actor_ownership` / `target_type`.
- Призванное существо не может атаковать в тот же раунд, в который оно вышло на стол.
- Защитные эффекты должны резолвиться раньше атакующих, иначе защита ощущается нечестной.
- Разрешаем несколько действий за раунд, но баланс контролируется через ресурсы и ограничения очереди, а не через логику UI.
- Draft валидируется авторитетно в `game-core` на lock текущего раунда.
- Если intent невозможен уже на snapshot начала раунда, lock отклоняется как `rejected`.
- Если intent был валиден на старте, но цель пропала по ходу резолва, действие остаётся в истории как `fizzled`.
- Любая спорная логика вида:
  - “что сработает раньше”;
  - “успевает ли бафф усилить атаку”;
  - “умирает ли цель до применения эффекта”;
  должна описываться как правило `game-core`, а не как договорённость во frontend.
- В текущий базовый срез уже входят:
  - массовые эффекты (`appliesToAllEnemies`, `appliesToAllCreatures`);
  - модификаторы `ignoreShield` и `ignoreEvade`;
  - повтор эффекта на следующий ход (`repeatNextTurn`);
  - control-эффекты уровня `cannot evade`, `skip action`, interrupt медленного заклинания и offensive trap;
  - безопасные `modifier/art` meta-rules, которые меняют следующее заклинание или следующую атаку, стоимость, скорость или восстановление ресурсов без перестройки общего round pipeline.

### 6.5. Текущий WebSocket-контракт PvP

- Публичные клиентские сообщения:
  - `join`
  - `roundDraft.replace`
  - `roundDraft.lock`
- Публичные серверные сообщения:
  - `state`
  - `roundDraft.snapshot`
  - `roundDraft.accepted`
  - `roundStatus`
  - `roundResolved`
  - `error`
- Legacy `action` больше не является частью публичного PvP transport.

### 6.6. Что ещё осталось после базовой миграции

- добить rule-completeness для edge cases и cleanup/end-of-round;
- расширить UI на все action kinds;
- расширить уже внедрённые structured reason/reject contracts на оставшиеся UX-кейсы по мере появления;
- развить уже внедрённый reveal timeline поверх `roundResolved` в staged-resolve UX:
  - центральная зона должна показывать раунд как последовательность слоёв резолва, а не как сырой лог;
  - каждый этап раскрывает только карты и эффекты текущего слоя;
  - между этапами допустим короткий прогресс-таймер/полоса перехода;
  - после последнего этапа UI уже переводит стол в новый раунд, обновляя руку, ману и пост-раунд состояние;
- синхронизировать остальные docs с фактической реализацией по мере развития PvP.
- сложные pipeline-breaking card mechanics не форсировать в текущий round engine:
  - условные эффекты карт;
  - status-слой вроде `burn`;
  - displacement/отталкивание;
  - эффекты, копирующие или перестраивающие уже собранный action pipeline.
- такие механики оформляются как отдельный future backlog, а не как implicit обещание каталога карт.
