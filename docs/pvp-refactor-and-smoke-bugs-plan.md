# PvP refactor and smoke bugs plan

## Цель

Зафиксировать текущий результат безопасного рефакторинга PvP-экрана и отдельно описать найденные smoke-баги. Дальше правки делаем пакетами: один пакет изменений -> кодовые проверки -> ручной smoke пользователем.

Важно: сейчас не продолжаем рефакторинг ради рефакторинга. Сначала закрываем визуальные и поведенческие проблемы, которые были найдены после smoke-проверки.

## Что уже сделали

Основной результат: `PlayPvpPage.tsx` перестал быть единственным местом для всей презентационной логики. Часть UI и вычислений вынесена в отдельные компоненты и хуки, при этом игровую механику не переносили из `game-core` и не меняли правила матча.

Вынесены компоненты PvP-экрана:

- `AuthRequiredPanel.tsx`
- `ExitConfirmDialog.tsx`
- `LocalBattleRibbon.tsx`
- `LocalHandTray.tsx`
- `MatchFeedDrawer.tsx`
- `MatchWaitingPanel.tsx`
- `OpponentPreparationZone.tsx`
- `PlayerSideCard.tsx`
- `ResolutionReplayStrip.tsx`
- `RoundDraftRejectedPanel.tsx`
- `SceneAlerts.tsx`
- `SceneInspectPanel.tsx`
- `SceneTopBar.tsx`
- `TurnActionRail.tsx`

Вынесены вспомогательные модули:

- `playback.ts`
- `presentation.ts`
- `selectors.ts`
- `serviceEvents.ts`
- `useMatchFeedDrawer.ts`
- `useResolvedReplay.ts`
- `useSceneInspect.ts`

Также были небольшие безопасные правки вне PvP:

- `frontend/src/main.tsx`: убран рискованный `root!`, добавлена явная проверка DOM root.
- `frontend/src/services/authSession.ts`: вынесена общая работа с auth-session.
- `frontend/src/services/authService.ts` и `frontend/src/services/api/axiosInstance.ts`: используют общий helper.
- `game-core/src/actions/AttackActionCommand.ts`: убран non-null assertion вокруг существа.
- `server/src/transport/ws/WsGateway.ts`: invite id переведен на более явный `randomUUID().slice(0, 8)`.

## Что удалось получить

- `PlayPvpPage.tsx` стал меньше и понятнее по зонам ответственности.
- UI зоны PvP теперь можно чинить точечно: replay, match feed, hand tray, side cards, inspect panel, battle ribbon.
- Поведение страницы проще тестировать, потому что крупные куски разнесены по файлам.
- После рефакторинга проходили проверки:
  - `npm.cmd --prefix frontend run type-check`
  - `npm.cmd --prefix frontend run test -- PlayPvpPage.test.tsx`
  - `npm.cmd run lint`
  - `npm.cmd --prefix frontend run build`

## Найденные проблемы

Ниже не утверждаем заранее, что каждый пункт появился именно после рефакторинга. Часть может быть регрессией, часть могла существовать раньше и стала видна только на smoke.

### 1. Deck/карточная зона растягивает или съедает блок персонажа

Симптомы:

- Если карт/значений много, scroll визуально залезает на область персонажа.
- Если карт нет, область становится слишком высокой и занимает лишнее пространство.

Где смотреть:

- `frontend/src/pages/DeckPage/DeckPage.tsx`
- `frontend/src/pages/DeckPage/DeckPage.module.css`
- связанные компоненты карточек/архетипа, если используются отдельно

Предварительное решение:

- Ограничить высоту зоны списка карт.
- Сделать пустое состояние фиксированным/умеренным по высоте.
- Развести область персонажа и scroll-область так, чтобы scrollbar не накладывался на информацию.

Риск: низкий, если правка только layout/CSS.

Smoke:

- много карт в списке;
- пустой список;
- широкий desktop viewport;
- проверить, что персонаж не обрезается и не перекрывается scrollbar.

### 2. Replay после resolve зажат вниз и плохо читается

Симптомы:

- Replay/лента резолва оказывается слишком низко.
- Карточки replay скрываются внутри сцены.
- Раньше визуально логичнее читалось сверху.

Где смотреть:

- `frontend/src/pages/PlayPvpPage/ResolutionReplayStrip.tsx`
- `frontend/src/pages/PlayPvpPage/useResolvedReplay.ts`
- `frontend/src/pages/PlayPvpPage/PlayPvpPage.module.css`
- layout сцены в `PlayPvpPage.tsx`

Предварительное решение:

- Вернуть replay в предсказуемую верхнюю/центральную readable-зону.
- Задать стабильную высоту и отступы для strip.
- Не давать replay попадать под нижнюю руку игрока или внутрь обрезаемого контейнера.

Риск: низкий/средний, потому что это presentation, но затрагивает главный layout матча.

Smoke:

- сыграть раунд с 2-4 картами;
- дождаться resolve;
- проверить, что replay читается без вертикального page-scroll.

### 3. Есть подозрение на сломанный resolve

Симптомы:

- По одному из скриншотов resolve выглядит некорректно, но нужно отделить визуальное сжатие replay от реальной логической ошибки.

Где смотреть:

- сначала frontend presentation:
  - `useResolvedReplay.ts`
  - `ResolutionReplayStrip.tsx`
  - `playback.ts`
  - `selectors.ts`
- только если данные реально неверные:
  - `game-core/`
  - server orchestration в `server/`

Предварительное решение:

- Сначала проверить, какие события и frames приходят в UI.
- Если данные корректные, чинить только отображение.
- Если данные некорректные, отдельно заводить пакет по `game-core`/server.

Риск: средний/высокий, если понадобится менять логику resolve. Пока это пункт расследования.

Smoke:

- повторить сценарий со скрина;
- сравнить фактический результат раунда с тем, что показывает replay.

### 4. История матча не восстанавливается после reconnect

Симптомы:

- После переприсоединения игрока match history пустая или неполная.
- Похоже, история сейчас частично собирается на фронте и не подгружается из состояния матча.

Где смотреть:

- `frontend/src/pages/PlayPvpPage/MatchFeedDrawer.tsx`
- `frontend/src/pages/PlayPvpPage/serviceEvents.ts`
- `frontend/src/pages/PlayPvpPage/useResolvedReplay.ts`
- server session/match state в `server/src/`
- существующие документы про match feed/storage, если они есть

Предварительное решение:

- Разобраться, где сейчас источник истины для истории.
- Если история нужна после reconnect, источник должен быть server/session state, а frontend должен только отображать.
- Добавить восстановление истории в snapshot/rejoin payload, если его нет.

Риск: средний/высокий, потому что может затронуть контракт WS и тесты сервера.

Smoke:

- сыграть 1-2 раунда;
- обновить страницу или переподключиться;
- проверить, что история раундов сохранилась.

### 5. Match feed: свернутые раунды занимают слишком много высоты

Симптомы:

- В одном состоянии раунды компактные.
- В другом состоянии карточки раундов растягиваются на всю доступную высоту.
- Пользователь хочет стабильный вид как на скрине с компактной ограниченной высотой.

Где смотреть:

- `frontend/src/pages/PlayPvpPage/MatchFeedDrawer.tsx`
- `frontend/src/pages/PlayPvpPage/PlayPvpPage.module.css`

Предварительное решение:

- Убрать растягивание collapsed cards.
- Задать компактную высоту collapsed state.
- Scroll держать только внутри списка истории, не внутри каждой карточки без необходимости.

Риск: низкий.

Smoke:

- 1 раунд;
- 3+ раунда;
- свернутые и раскрытые состояния;
- проверить, что drawer не ломает высоту сцены.

### 6. Призванное существо улетает вверх

Симптомы:

- После призыва `Искровой дух` карточка существа оказывается в верхней части поля вместо стабильной зоны.

Где смотреть:

- `frontend/src/pages/PlayPvpPage/LocalBattleRibbon.tsx`
- `frontend/src/pages/PlayPvpPage/PlayPvpPage.module.css`
- selectors board state в `selectors.ts`

Предварительное решение:

- Зафиксировать зоны для существ игрока и оппонента.
- Убрать зависимость позиции существа от высоты replay/руки/других динамических блоков.
- Проверить, не тянет ли flex/grid карточку в неожиданный край.

Риск: средний, потому что влияет на основную боевую сцену.

Smoke:

- призвать 1 существо;
- призвать 2+ существа;
- проверить desktop viewport без page-scroll.

### 7. На столе оппонента не отображается то, что он выложил после resolve

Симптомы:

- После resolve игрок должен знать раскрытые карты/существа оппонента.
- На столе оппонента это не отображается.

Где смотреть:

- `frontend/src/pages/PlayPvpPage/OpponentPreparationZone.tsx`
- `frontend/src/pages/PlayPvpPage/selectors.ts`
- board/rendering в `PlayPvpPage.tsx`
- server snapshot, если frontend не получает нужное состояние

Предварительное решение:

- Проверить, есть ли раскрытые карты/существа оппонента в client state после resolve.
- Если данные есть, добавить корректный render в opponent board zone.
- Если данных нет, проверить server payload/snapshot.

Риск: средний, потому что это может быть только render, а может быть контракт состояния.

Smoke:

- оппонент призывает существо;
- resolve завершен;
- локальный игрок видит раскрытое существо/карту на стороне оппонента.

### 8. Нет выбора цели для урона при наличии мага и существа

Симптомы:

- При `Вспышке пламени` и наличии существа у оппонента нет выбора: ударить мага или существо.

Где смотреть:

- card definition для `Вспышка пламени`
- target helpers в `PlayPvpPage.tsx`
- `LocalBattleRibbon.tsx`
- `SceneInspectPanel.tsx`
- `game-core/`, только если ошибка в типах целей или правилах

Предварительное решение:

- Проверить, какой `targetType` у карты.
- Проверить, возвращает ли UI candidates для мага и существ.
- Если карта по дизайну должна бить разные типы целей, привести definition/UI к одному контракту.

Риск: средний/высокий, потому что может затронуть правила выбора целей.

Smoke:

- у оппонента есть маг и существо;
- выбрать карту урона;
- увидеть оба валидных target candidates;
- применить по магу и отдельно по существу.

### 9. Действия существа раздувают карточку

Симптомы:

- При раскрытии существа внутри карточки появляются большие блоки `режим`, `текущая цель`, `действие`, `сброс`.
- Карточка становится слишком высокой, текст плохо читается, layout ломается.

Где смотреть:

- `frontend/src/pages/PlayPvpPage/LocalBattleRibbon.tsx`
- `frontend/src/pages/PlayPvpPage/SceneInspectPanel.tsx`
- `frontend/src/pages/PlayPvpPage/useSceneInspect.ts`
- `frontend/src/pages/PlayPvpPage/PlayPvpPage.module.css`

Предварительное решение:

- Карточка существа должна иметь фиксированную высоту.
- На самой карточке оставить только компактные статусы/иконки/кнопки.
- Подробные действия перенести в inspect panel или отдельную компактную панель выбора действия.
- Убрать внутренние scroll внутри карточки существа.

Риск: средний, потому что это UX текущего выбора действия.

Smoke:

- выбрать существо;
- выбрать цель;
- сменить действие;
- сбросить цель;
- убедиться, что карточка не растягивается.

### 10. Несколько существ/карт растягивают все карточки

Симптомы:

- При появлении еще одной карты/существа меняется высота или размер всех карточек.
- Layout становится непредсказуемым.

Где смотреть:

- `frontend/src/pages/PlayPvpPage/LocalBattleRibbon.tsx`
- `frontend/src/pages/PlayPvpPage/PlayPvpPage.module.css`

Предварительное решение:

- Зафиксировать размеры карточек существ.
- Использовать wrap/compact layout без изменения высоты карточек.
- Вторичные данные показывать бейджами и иконками.

Риск: низкий/средний.

Smoke:

- 1 существо;
- 2 существа;
- 3+ существа;
- проверить, что карточки остаются одинаковой высоты.

## Пакеты работ

### Пакет 0. Baseline перед исправлениями

Что делаем:

- Фиксируем текущий commit и список известных симптомов.
- Не меняем код.
- Дальше каждый пакет проверяем отдельно.

Результат:

- Есть этот документ как точка синхронизации.

### Пакет 1. Быстрые layout-исправления без игровой логики

Входит:

- Deck/карточная зона: ограничить высоты, убрать наложение scroll на персонажа.
- Match feed: сделать collapsed rounds компактными и стабильными.
- Replay strip: вернуть читаемое положение и стабильную высоту.

Ожидаемый риск: низкий.

Кодовые проверки:

- `npm.cmd --prefix frontend run type-check`
- `npm.cmd --prefix frontend run test -- PlayPvpPage.test.tsx`
- `npm.cmd --prefix frontend run build`

Ручной smoke:

- deck с пустым и заполненным списком;
- PvP resolve replay;
- match feed с 1 и 3+ раундами.

### Пакет 2. Стабилизация board layout

Входит:

- Призванные существа не улетают вверх.
- Карточки существ имеют фиксированную высоту.
- Несколько существ не растягивают друг друга.
- Большие inline-действия существа убираются из карточки.

Ожидаемый риск: средний.

Кодовые проверки:

- `npm.cmd --prefix frontend run type-check`
- `npm.cmd --prefix frontend run test -- PlayPvpPage.test.tsx`
- `npm.cmd --prefix frontend run build`

Ручной smoke:

- призыв одного существа;
- призыв нескольких существ;
- выбор существа и действия;
- проверка читаемости без page-scroll.

### Пакет 3. Видимость состояния оппонента после resolve

Входит:

- Проверить, приходит ли раскрытое состояние оппонента в frontend.
- Отобразить известные карты/существа оппонента на его стороне поля.
- Если данных нет в snapshot, отдельно отметить необходимость server-контракта.

Ожидаемый риск: средний.

Кодовые проверки:

- frontend проверки;
- server/game-core тесты только если придется менять контракт или состояние.

Ручной smoke:

- оппонент призывает существо;
- после resolve локальный игрок видит это существо на стороне оппонента.

### Пакет 4. Target selection для урона

Входит:

- Проверить definition `Вспышка пламени`.
- Проверить target candidates для мага и существ.
- Исправить UI или contract, чтобы валидные цели выбирались явно.

Ожидаемый риск: средний/высокий.

Кодовые проверки:

- frontend тесты по PlayPvpPage;
- game-core тесты, если меняются правила цели.

Ручной smoke:

- у оппонента есть маг и существо;
- карта урона предлагает обе цели;
- урон корректно применяется к выбранной цели.

### Пакет 5. История матча после reconnect

Входит:

- Найти источник истины для match history.
- Перестать полагаться только на frontend-сборку истории, если сейчас так происходит.
- Добавить восстановление истории через server snapshot/rejoin payload, если его нет.

Ожидаемый риск: высокий относительно UI-only правок, потому что может затронуть WS/server contract.

Кодовые проверки:

- frontend tests;
- server WS tests;
- build.

Ручной smoke:

- сыграть 1-2 раунда;
- обновить страницу/переподключиться;
- история раундов сохраняется и отображается.

## Что пока не делаем

- Не продолжаем дробить `PlayPvpPage.tsx`, пока не стабилизируем найденные баги.
- Не меняем `game-core`, пока не докажем, что проблема не только в отображении.
- Не трогаем deploy, env, docker, CORS, service URLs.
- Не делаем крупный redesign всего PvP-экрана одним коммитом.

## Правило движения дальше

1. Берем один пакет.
2. Делаем минимальный набор правок.
3. Запускаем кодовые проверки.
4. Пользователь делает ручной smoke по списку.
5. Только после этого идем к следующему пакету.
