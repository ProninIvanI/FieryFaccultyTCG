  - сначала уточняем контекст в репозитории (где код, какие команды запуска);

## DEPLOY SAFETY

- Изменения в `docker-compose*`, `Dockerfile*`, `.env*`, CORS, портах, service URL, auth/internal token и Postgres-конфиге считаются high-risk.
- Для VPS/staging источником истины являются `docker-compose.staging.yml`, `.env.staging.example`, `docs/staging-vps.md` и `DEPLOY.md`; локальный `docker-compose.yml` не считается deploy-конфигом по умолчанию.
- Нельзя переносить локальные дефолты (`localhost`, bind mounts, dev-only команды, Vite dev server) в staging/deploy без явной проверки server runtime.
- Если меняются frontend fallback URL или service endpoints, нужно отдельно проверить:
  - локальный запуск;
  - staging/docker-run;
  - доступ по реальному IP/домену.
- Внутри docker-сети сервисы должны ходить по именам контейнеров (`backend`, `postgres`), а не по `localhost`.
- Нельзя менять deploy startup (`node dist`, `tsx`, `vite preview`, и т.п.) без проверки, что текущий build output реально стартует в контейнере.
- После infra/deploy-правок нужно синхронизировать `docker-compose.staging.yml`, `.env.staging.example`, `docs/staging-vps.md` и `DEPLOY.md`, чтобы инструкция и конфиг не расходились.
- После infra/deploy-правок обязательны smoke-checks: `health`, login, deck list/save, PvP connect.
- Перед рекомендацией `docker compose down -v` обязательно явно предупредить, что команда удалит данные Postgres volume.
- Нельзя рекомендовать смену `POSTGRES_PASSWORD` на уже поднятом стенде без пояснения, что существующий volume сохранит старый пароль.
- Перед рискованными действиями с БД нужно сначала предложить backup и только потом destructive reset.

## CONSTRAINTS

- Любая логика, влияющая на исход матча (рандом, эффекты, резолв событий), реализуется только в `game-core/`.
- Сервер и фронтенд не должны добавлять рандом, влияющий на игровой state.
- Любые сообщения по сети (WebSocket/HTTP) валидируются и типизируются; внешним данным не доверяем.
- В логах и в репозитории не должно быть секретов и данных из `.env`.

## VISUAL UX GUARDRAILS

- Игровая зона активного матча является first-class UX-сценой: на desktop она не должна вылезать за viewport по высоте в нормальном боевом состоянии.
- Для PvP и других игровых экранов целевой baseline: без вертикального page-scroll после входа в матч и без необходимости «летать» вверх-вниз ради чтения стола.
- Если матчевый экран не помещается по высоте, сначала уменьшаем плотность UI:
  - ужимаем отступы, служебные панели, декоративные блоки, размеры карт и вторичную типографику;
  - переводим вспомогательные элементы в compact/hud-режим;
  - и только потом рассматриваем локальный scroll внутри вторичных зон.
- Ключевые элементы матча должны быть видны одновременно без прокрутки:
  - статус/фаза раунда;
  - игровое поле;
  - рука локального игрока;
  - основные кнопки текущего решения.
- Внутренний scroll в боевых зонах не должен быть решением по умолчанию. Если scroll всё же нужен, он должен быть локальным, ожидаемым и не резать карточки/контент по краям.
- Карты, бейджи и интерактивные элементы не должны визуально обрезаться контейнером в базовом состоянии.
- Контраст и читаемость обязательны: не использовать слабочитаемые сочетания вроде жёлтого текста на сером фоне; декоративный стиль не важнее чтения состояния матча.
- PvP, Deck Builder и другие карточные экраны должны держать визуально согласованный язык: близкие принципы плотности, типографики, контраста и поведения карточек.
- Любое заметное изменение layout/плотности игрового экрана требует ручной smoke-проверки прямо в браузере как минимум для desktop viewport.
- Визуальную ручную smoke-проверку выполняет пользователь. AI-агент не запускает browser smoke/скриншоты для оценки внешнего вида без явной просьбы; вместо этого он выполняет кодовые проверки (`lint`, `type-check/typecheck`, тесты, сборка) и кратко отмечает, что именно пользователю стоит посмотреть глазами.
