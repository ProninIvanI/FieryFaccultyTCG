# Архитектура Backend

Проект использует слоистую архитектуру (Layered Architecture) для обеспечения масштабируемости и поддерживаемости кода.

## Структура папок

```
src/
├── config/          # Конфигурация приложения (БД, настройки)
├── routes/          # Определение маршрутов API
├── controllers/     # Обработка HTTP запросов и ответов
├── services/        # Бизнес-логика приложения
├── models/          # Модели данных (entities/interfaces)
├── middlewares/     # Express middleware (валидация, обработка ошибок)
├── utils/           # Вспомогательные утилиты
├── types/           # TypeScript типы и интерфейсы
└── index.ts         # Точка входа приложения
```

## Описание слоев

### 1. **config/** - Конфигурация
Содержит настройки приложения:
- Подключение к базе данных
- Настройки окружения (переменные окружения)
- Конфигурация внешних сервисов

**Примеры:**
- `env.ts` - централизованная конфигурация переменных окружения с валидацией
- `database.ts` - настройка пула подключений PostgreSQL
- `index.ts` - экспорт всей конфигурации

**Конфигурация окружения (`env.ts`):**
- Централизованное управление переменными окружения
- Валидация при запуске приложения
- Типизированные значения по умолчанию
- Разделение на секции: server, database, api, cors
- Автоматическое формирование connection string для БД

### 2. **routes/** - Маршруты
Определяет эндпоинты API и связывает их с контроллерами.

**Принципы:**
- Один файл роутов на одну сущность/домен
- Минимальная логика, только маршрутизация
- Импорт контроллеров для обработки запросов

**Примеры:**
- `index.ts` - главный роутер, объединяет все роуты
- `healthRoutes.ts` - роуты для health check

### 3. **controllers/** - Контроллеры
Обрабатывает HTTP запросы и ответы.

**Принципы:**
- Валидация входных данных
- Вызов сервисов для бизнес-логики
- Формирование HTTP ответов
- Обработка ошибок на уровне HTTP

**Примеры:**
- `healthController.ts` - обработка health check запросов

### 4. **services/** - Сервисы (Бизнес-логика)
Содержит основную бизнес-логику приложения.

**Принципы:**
- Независимы от HTTP слоя
- Могут использовать модели и утилиты
- Могут вызывать другие сервисы
- Работа с базой данных через модели/repositories

**Примеры:**
- `healthService.ts` - проверка состояния системы

### 5. **models/** - Модели данных
Определяет структуру данных приложения.

**Принципы:**
- Интерфейсы/типы для TypeScript
- Модели для работы с БД (если используется ORM)
- Валидация структуры данных

**Примеры:**
- Интерфейсы для User, Product и т.д.

### 6. **middlewares/** - Middleware
Express middleware для обработки запросов.

**Принципы:**
- Переиспользуемые функции
- Обработка ошибок
- Валидация
- Аутентификация/авторизация
- Логирование

**Примеры:**
- `errorHandler.ts` - глобальная обработка ошибок
- `notFoundHandler.ts` - обработка 404 ошибок

### 7. **utils/** - Утилиты
Вспомогательные функции, используемые во всем приложении.

**Принципы:**
- Чистые функции (pure functions)
- Без побочных эффектов
- Переиспользуемость

**Примеры:**
- `logger.ts` - логирование
- Функции форматирования, валидации и т.д.

### 8. **types/** - Типы TypeScript
Общие типы и интерфейсы для всего приложения.

**Принципы:**
- Переиспользуемые типы
- Типы для API ответов
- Типы для конфигурации

**Примеры:**
- `ApiResponse<T>` - стандартный формат ответа API
- `HealthCheckResponse` - тип для health check

## Поток данных

```
HTTP Request
    ↓
routes/          (определение маршрута)
    ↓
controllers/     (обработка HTTP, валидация)
    ↓
services/        (бизнес-логика)
    ↓
models/          (работа с данными)
    ↓
config/database  (запросы к БД)
    ↓
HTTP Response
```

## Пример добавления новой функциональности

Допустим, нужно добавить CRUD для пользователей:

1. **types/index.ts** - добавить интерфейс `User`
2. **models/index.ts** - добавить модель `UserModel`
3. **services/userService.ts** - бизнес-логика (создание, чтение, обновление, удаление)
4. **controllers/userController.ts** - обработка HTTP запросов
5. **routes/userRoutes.ts** - определение маршрутов
6. **routes/index.ts** - подключить `userRoutes`

## Преимущества такой архитектуры

✅ **Масштабируемость** - легко добавлять новые функции  
✅ **Поддерживаемость** - четкое разделение ответственности  
✅ **Тестируемость** - каждый слой можно тестировать отдельно  
✅ **Переиспользование** - сервисы и утилиты можно использовать в разных местах  
✅ **Читаемость** - понятная структура для новых разработчиков
