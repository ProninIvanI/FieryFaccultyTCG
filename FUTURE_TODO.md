Definition of Playable для ближайшей версии:
- два игрока могут войти в один матч без ручных правок state;
- игроки видят не только raw snapshot, но и базовое игровое поле;

- действия одного игрока синхронно отражаются у второго;
- сценарий `login -> join PvP -> action -> EndTurn` воспроизводим вручную и не считается разовой удачей;
- quality gates и smoke-проверки остаются зелёными или предсказуемо контролируемыми.

## План По Архитектуре Данных

Контекст и выводы:
- Сейчас пользовательские данные живут во `frontend/localStorage`, а не в `backend + postgres`.
- Такой подход подходит только для локального demo/mock-сценария и мешает нормальной серверной идентичности.
- Для проекта нужно явно зафиксировать `source of truth` для каждого типа данных, иначе данные будут размазаны между `frontend`, `backend`, `server` и `game-core`.
- `frontend` должен хранить только UI-state и короткоживущую клиентскую сессию, но не список пользователей и не матчевые данные как системную истину.
- `backend + postgres` должны стать источником истины для пользователей, контента, колод, истории матчей и прочих персистентных сущностей.
- `server` должен хранить только runtime-состояние активных матчей и связь `socket -> authenticated user/player`.
- `game-core` остаётся источником истины для детерминированной игровой логики и доменной модели матча.

Целевая схема хранения по доменам:
- Пользователи, auth/session metadata, профили, настройки аккаунта -> `backend + postgres`
- Колоды игроков и связанные пользовательские сущности -> `backend + postgres`
- Новости, админские настройки, debug/admin-конфиг -> `backend + postgres`
- Активные PvP-сессии, runtime-state матча, сокеты игроков -> `server` в памяти
- История матчей, replay, результаты, журнал команд -> `backend + postgres`
- Карты, персонажи и прочие game-data справочники -> пока data-driven файлы в репозитории, с возможным переносом в БД позже при появлении админского редактирования
- UI-состояние и клиентские предпочтения -> `frontend`, локально, только если они не влияют на shared/server state

Риски текущего состояния:
- Пользователи не общие между браузерами и устройствами.
- `playerId` для PvP пока берётся из локальной browser-only auth.
- Нельзя нормально развивать серверную идентичность, аудит и восстановление сессий.
- Сложно разделить demo/mock и реальные пользовательские данные.

Что хотим получить в ближайшей целевой версии:
- Один понятный `source of truth` на каждый тип данных.
- Пользователи и login/register живут на backend.
- `frontend` больше не хранит список пользователей в `localStorage`.
- `WS player identity` опирается на backend-auth, а не на произвольный `userId` из UI.
- Появляется понятная база для будущих колод, матчей, replay и админки.
